3. Security Findings

Critical All order tables readable by anyone

The RLS policies on report_orders, checklist_orders, and quote_review_orders have SELECT with qual: true. Anyone with the public anon key (visible in browser JS) can read every customer's name, email, pool address, and payment details.

Fix: Drop the "Allow anon read by id" SELECT policies on all three tables. No client code reads from them.

Critical Checkout route has no ownership verification

Anyone who knows an orderId can initiate a checkout session for that order, leaking the customer's email address. Combined with the open SELECT policy above, attackers can enumerate all orders.

Fix: Require the customer's email in the checkout request and verify it matches the order.

Critical Upload routes accept any orderId

The upload-url and confirm-upload API routes accept any orderId with no ownership check. An attacker can attach files to or overwrite photos on any existing order.

Fix: Require email verification against the order record before generating upload URLs.

Critical contact-uploads bucket is public with 50MB limit

The Supabase storage bucket contact-uploads is publicly readable and writable. Can be abused as free file hosting or to store malicious content, running up storage costs.

Fix: Make the bucket private, serve files through signed URLs.

Critical Next.js 14.2.35 has 4 known vulnerabilities

Including HTTP request smuggling (GHSA-ggv3-7p47-pfv8) and DoS via React Server Components (GHSA-h25m-26qc-wcjf). These are not fully patched in any 14.x release.

Fix: Upgrade to Next.js 15.x (breaking changes - test thoroughly).

High HMAC token truncated to 128 bits

Download tokens use .slice(0, 32) on hex output, halving the HMAC output space. Not immediately exploitable but unnecessary weakness.

Fix: Use full HMAC output. Note: breaks existing download links.

High No email format validation on any API route

Email fields accept any string. Can waste Brevo quota and damage sender reputation with bounces.

Fix: Add server-side email regex validation on all routes.

High No CSRF protection

All POST routes accept any origin. An attacker could craft forms on other sites to spam endpoints.

High Stripe success_url uses request Origin header

An attacker could set Origin: https://evil.com to redirect post-payment to a phishing page.

Fix: Hardcode origin to https://fibreglasspoolcheck.com.au.

Medium Rate limiter is in-memory, ineffective on Vercel

Each serverless invocation gets a fresh Map. Rate limiting is essentially non-functional in production.

Medium DOWNLOAD_TOKEN_SECRET missing from .env.local.example

The code throws if this env var is missing, but it's not documented in the example file.

Low Fallback token in red-flags-guide route

If DB insert fails, all download links share the same HMAC input ('fallback-token'). One valid link works for everyone.

Fix: Use crypto.randomUUID() instead of static fallback.

What's Done Well

  • Clean separation of concerns - components, lib, API routes
  • Stripe webhook signature verification correctly implemented
  • HTML escaping on all user inputs in emails via escapeHtml()
  • HMAC-SHA256 with timing-safe comparison for download tokens
  • Email failures don't break the payment webhook
  • Error boundary, loading state, and 404 page all present
  • Proper Next.js Image component with alt tags throughout
  • Semantic HTML with proper heading hierarchy
  • ~1,600 LOC across 16 files - lean and maintainable

Issues Found

High Footer uses class instead of className

Footer.jsx line 55 - styles silently don't apply, React logs a console warning.

High Hardcoded absolute URLs in PoolCheckReport

Two Link components use https://fibreglasspoolcheck.com.au/... instead of relative paths. Breaks dev/preview and bypasses client-side navigation.

High Orphaned quote_review_orders table

Table exists in Supabase with 3 rows and active RLS policies, but the feature was removed from the site. Anon INSERT is still enabled.

Medium No favicon

Missing from /public/ and /app/. Browsers 404 on every page load.

Medium Inconsistent messaging: "7 warning signs" vs "10 warning signs"

BuyingAHouse page says 7, RedFlagsGuide page says 10. One is wrong.

Medium Unused remote image patterns

cdn.pixabay.com and media.base44.com in next.config.js are never used. Leftover attack surface.

Medium Missing canonicals and breadcrumbs on some pages

RedFlagsGuide missing canonical URL. BuyingAHouse and PoolOwner missing BreadcrumbSchema.

Medium BuyerChecklist schema uses relative URL

Structured data URLs must be absolute for Google to parse them.

Low Google Ads conversion labels may be wrong

send_to values use human-readable names (red_flags_guide) instead of Ads-format labels. Conversion tracking may not be working.

Low Sitemap uses new Date() for all entries

Every crawl shows today's date. Blog posts should use actual publication dates.

Low Unused Tailwind brand color tokens

brand.dark and brand.orange defined but never referenced.

Why Google Isn't Indexing

Likely causes (in order of probability):

  1. Site is very new (launched ~19 March 2026) - Google hasn't crawled enough yet
  2. Sitemap was not successfully submitted to GSC (quota exceeded 19 March)
